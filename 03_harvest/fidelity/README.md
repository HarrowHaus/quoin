# Fidelity extraction framework

Three-method pipeline for upgrading harvested token packs from
designed-approximation to byte-faithful extraction. Built in Phase 3.5
and extended in Phase 3.5b.

## Layout

```
fidelity/
├── extract.js         Orchestrator. Dispatches A/B/C per spec shape.
├── runner.js          Method-specific runners (runMethodA/B/C) + helpers.
├── parsers.js         Format parsers: scss, css-vars, js, ts, dtcg,
│                      yaml, style-dictionary, primer-json5, value-wrapped-ts.
├── oklch.js           culori wrappers; canonical `oklch(L% C H)` output.
├── annotate-tier-c.js Idempotent Tier C stamper for unspecced packs.
├── specs/             Per-pack extraction specs (one .js per pack).
├── canonical/         Cached upstream source text for verification.
└── extract-report.json  Per-run outcome (tier, method, value count).
```

## The three methods

### Method A — static fetch + parse

For systems with canonical static token files (CSS, SCSS, TS, JSON,
YAML, etc).

```js
// fidelity/specs/<name>.js
export default {
  fetch: {
    urls: ["https://raw.githubusercontent.com/.../tokens.css"],
    format: "css-vars",   // PARSERS key
    commit: "main"
  },
  map(values) {
    return { base: { ... }, notes: "..." };
  }
};
```

Available parsers: `scss`, `css-vars`, `js`, `ts`, `dtcg`, `yaml`,
`style-dictionary`, `primer-json5`, `value-wrapped-ts`.

### Method B — algorithm execution

For systems whose canonical form is a generation algorithm
(Material 3, Ant Design), or where the source lib exports static
values that aren't conveniently fetchable.

**Authorized to `npm install`** the source-system library as a dev
dependency of the harvest pipeline. The published Quoin pack ships
only the static OKLCH values the algorithm produced — the algorithm
is never embedded in the pack.

```js
// fidelity/specs/<name>.js
import { themeFromSourceColor, argbFromHex, hexFromArgb }
  from "@material/material-color-utilities";

export default {
  async algorithm() {
    const theme = themeFromSourceColor(argbFromHex("#6750A4"));
    const out = {};
    for (const tone of [0, 10, ..., 100]) {
      out[`primary.${tone}`] = hexFromArgb(theme.palettes.primary.tone(tone));
    }
    return {
      rawValues: out,
      library: "@material/material-color-utilities",
      version: "0.2.7",
      inputs: { seed: "#6750A4" }
    };
  },
  map(values) { /* ... */ }
};
```

The runner records `library@version` and `inputs` in
`attribution.harvestNotes` so consumers can regenerate.

### Method C — per-file structured extraction

For systems whose values are static but split across multiple files
in patterns that defeat concatenated parsing (MUI per-colour modules,
Orbit per-family palette JSON). Each file fetched and parsed
independently with file-derived namespacing.

```js
// fidelity/specs/<name>.js
export default {
  files: [
    { name: "grey", namespace: "grey",
      url: "https://.../grey.js", format: "js" },
    { name: "indigo", namespace: "indigo",
      url: "https://.../indigo.js", format: "js" }
  ],
  map(values) {
    // values has `grey.50`, `grey.100`, ..., `indigo.50`, ...
    return { base: { ... } };
  }
};
```

A spec can combine methods (e.g. Method B for the brand palette,
Method A for status colours). The runner merges flat
`{name: rawColor}` maps from each method before handing them to
`spec.map()`.

## Adding a new pack

1. Identify the source system's canonical token location. Try:
   - Official GitHub repo `/tokens`, `/styles`, `/theme`, or
     `/design-tokens` directories.
   - Published npm package (often `@org/tokens` or `@org/colors`).
   - The system's design site (m3.material.io,
     carbondesignsystem.com, etc.).
2. Pick a method:
   - Static file fetchable? **Method A.**
   - Tokens algorithmically generated by a published library? **Method B.**
   - Tokens split across per-family files? **Method C.**
3. Write `fidelity/specs/<name>.js`.
4. Add the parser to `parsers.js` if the format is novel.
5. Run `node fidelity/extract.js <name>` — verify tier outcome.
6. Run `node build.js && node validate.js` — confirm the pack still
   compiles cleanly.
7. Update `03_harvest/REPORT.md` with the per-pack row.
8. Commit `fidelity-b: extract <name> via method <A|B|C>`.

## Tier classification rules

The orchestrator stamps `attribution.fidelityTier`:

- **A** — extracted with no mapping interpretation. `spec.map()` returned
  no `notes`, and the existing semantic refs resolve against the new base.
- **B** — extracted with documented mapping decisions (notes returned,
  or some semantic-ref family was missing and a mapping was chosen).
- **C** — extraction failed. Spec missing, fetch failed, parser yielded
  no values, or algorithm/map error.

For Tier C packs, document the reason explicitly in
`attribution.harvestNotes`. Tier C should remain rare (≤3 in the
catalogue) — see `03_harvest/REPORT.md` for the documented unresolvable
list (currently: clarity, geist, workday).
