Accessibility checklist

  • Each <article class="pricing-tier"> uses aria-labelledby pointing to its tier-name heading so screen readers announce "Pro tier: $12 per user per month…" as a single coherent unit.
  • "Most popular" badge is decorative text in the visual; aria-label="Most popular plan" on the badge provides the screen-reader announcement.
  • Featured tier's visual elevation (translateY(-8px) on desktop) collapses on mobile to preserve tap-order rhythm.
  • Excluded features still appear in the bullet list as <li class="excluded"> with an X icon — text conveys exclusion, not just colour (the X icon is in --text-disabled, the list item itself is text-recede).
  • Comparison table uses semantic <table> with <thead> + <tbody>. Sticky thead during vertical scroll. Category subheaders are inside <tbody> using td colspan="5" with monospace label styling.
  • Em-dash (—) used in excluded cells, not "N/A" or empty string — explicit "this is not included" signal at the visual + screen-reader level.
  • Tabular numerics enforced on every price + price-mini + savings figure so digit-bearing values align mid-card-render.
  • Billing toggle is a real role="switch" + aria-checked (delegated to pattern-form-fields) OR role="radiogroup" + role="radio" (delegated to pattern-button-system's connected button-group register). Toggle change announces price update via aria-live="polite".
  • Checkout-initiated state: active tier's button uses aria-busy="true" + transparent label + spinner overlay (delegated to action-button's loading microstate). Other tier buttons disable to prevent double-submission. Live region announces redirect.
  • Selected state: aria-selected="true" on the article; the picked tier persists across multi-step checkout via this attribute.
  • 44×44 hit target on every tier CTA (delegated to action-button's min-height=44 floor at md size).
  • Reduced-motion: skeleton animation stops; featured-tier desktop-only translateY remains (it's a layout offset, not a transition).

Composition lineage — what this pattern consumes

First pattern in the catalog to compose from sibling pattern packs. Documenting the consumption chain explicitly here sets the convention for future sibling-composition patterns.

Consumed primitive From pack Used by this pattern as
<action-button> @quoin/pattern-button-system Every tier CTA — intent=primary on featured tier; intent=secondary on default + starter; intent=link on enterprise "Talk to sales"
<button-group> @quoin/pattern-button-system Segmented variant of billing-toggle (connected register, role=radiogroup)
<form-control type="switch"> @quoin/pattern-form-fields Switch variant of billing-toggle (role=switch + aria-checked)
<feature-cell>
(optional)		 @quoin/pattern-feature-grid Available below the tier-card grid for "compare all features" callout cells. Not consumed in default tier-card chrome (which uses plain <ul> + checkmarks for visual clarity).
canonical tokens @quoin/tokens-baseline Every CSS value reads from a canonical --var; zero hardcoded values past the :root seed.

The convention: pattern packs CAN compose from other pattern packs. When they do, the README documents the consumption chain explicitly so the composition is legible (this section is the prototype for that documentation pattern). Vocabulary primitives can also be consumed directly — pattern-form-fields composes from vocab-forms — but inter-pattern composition is the higher-leverage relationship at the catalog level.
