Composition lineage — what form-validation consumes

Consumed primitiveSource packUsed inHow this pattern uses it
<form-control> (all types) @quoin/pattern-form-fields Every form field This pattern adds validation on top of form-fields. aria-invalid, aria-required, aria-describedby all live on the form-control. The validation pattern only manages the orchestration + message rendering.
<form-section> + <form-row> @quoin/pattern-form-fields Form scaffold The form-section's error-summary slot is where <validation-summary> mounts. The form-section's submit-button slot composes button-system.
<action-button intent="primary"> @quoin/pattern-button-system Submit button (every form) Inherits 8 microstates. aria-busy="true" while form is validating or submitting. aria-disabled="true" while async validation completes pre-submit.
<action-button intent="secondary"> or intent="ghost" @quoin/pattern-button-system Secondary actions in success/error states (Resend, Retry, Cancel) De-escalated CTAs alongside the primary action in submit-failed and submitted states.
canonical tokens @quoin/tokens-baseline every CSS value Critical / warning / success colour families drive the four summary tones. role + aria-live + tabindex drive the semantic behaviour.

Accessibility checklist

  • Error summary as landmark: <section role="alert" aria-labelledby="..." tabindex="-1"> — programmatically focusable after a failed submit so screen readers immediately announce the count + contents.
  • Skip-to-error links: Each <li><a href="#field-id"> in the summary anchors to its field. Per WCAG 2.4.6 (Headings and Labels) + 3.3.1 (Error Identification).
  • Field-level error connection: Invalid field has aria-invalid="true" + aria-describedby="field-id-error [field-id-help]" pointing to the validation-message(s) by id.
  • role="alert" vs role="status": Error tone uses role="alert" (assertive, interrupts). Warning / info / success use role="status" + aria-live="polite" (non-interrupting).
  • Required field marker: Visible * next to label is aria-hidden="true"; the actual requirement is communicated via aria-required="true" on the field.
  • Live-validation announcements: Use aria-live="polite" on the message container so AT announces async results (e.g., "Checking username…" → "Available") without stealing focus.
  • Async validation status: Validating state is a third state (not invalid). Status icon uses status="validating"; field aria-invalid stays "false" or unset. aria-busy="true" on the field OR on the form.
  • Password strength meter: Uses role="progressbar" with aria-valuenow / aria-valuemin / aria-valuemax + a textual label. The label changes live in an aria-live="polite" region.
  • Constructive error messages: "Enter your email address" beats "Email is required" — the imperative form tells the user what to do, not what they failed at.
  • Focus management on failed submit: JS calls summary.focus() after preventDefault. Don't auto-focus the first invalid field — that skips the user past the summary they might want to read.
  • Submit button states: aria-busy="true" during async validation + submission. aria-disabled="true" when form is in validating state and user-initiated submit should not retrigger.
  • novalidate on form: Disables browser-native validation so this pattern owns all messaging. Required for cross-browser consistency.
  • Reduced motion: Spinner animations + strength-bar fill transition collapse to none under prefers-reduced-motion: reduce.
