Keep Tailwind's class generation; replace the long utility strings in your source with semantic tags. The compiled output is still Tailwind classes. You don't replace Tailwind. You replace authoring against Tailwind classes directly. Quoin's impl-tailwind pack emits Tailwind v4 classes for you. The shape of the change

In Tailwind, the unit of authorship is the utility class string. Every visual decision is encoded inline.

<!-- Tailwind --> <button class="px-4 py-2 bg-stone-900 text-white rounded-md font-medium hover:bg-stone-800"> Save </button> <h1 class="text-7xl font-serif font-medium tracking-tight text-stone-900"> The Bot Smell Problem </h1>

In Quoin, the unit of authorship is the semantic element. The same compiled output, but the source encodes intent rather than implementation.

What you keep

Tailwind v4's CSS generation, your existing tailwind.config, your colour palette, your build pipeline. @quoin/impl-tailwind emits Tailwind v4 arbitrary-value classes referencing CSS custom properties — bg-[var(--accent)], text-[var(--text-emphasis)], and so on. Tailwind picks them up on its normal content scan.

Your tokens become the canonical semantic-token namespace. --accent, --surface, --text-emphasis — a small, intentional set that every Quoin primitive references. See the token spec.

What changes

Source authoring. Code review. AI prompts. A <primary-action> is reviewable for intent; class="px-4 py-2 bg-stone-900 …" isn't. Reviewers can read intent in seconds instead of mentally parsing utility strings.

Token discipline. Quoin enforces a canonical semantic-token namespace. Stray utility strings that reach for bg-stone-700 without going through --text won't compile through Quoin primitives. Your palette becomes a contract.
